Deployment

Production deployment guidance: reverse proxy, TLS, base URLs, backups, and upgrades.

Recommended production shape

Network

Reverse proxy

Run TimeboardApp on an internal port (8888) and terminate TLS at Nginx/Caddy/Traefik.

Storage

Persistent volume

Mount /data to persist settings.yml and the SQLite database.

Secrets

Env overrides

Provide secrets via environment variables to keep them out of the repository.

Base URL (important)

If TimeboardApp is served behind a reverse proxy (for example at https://tasks.example.com), set app.base_url (or TIMEBOARDAPP_BASE_URL) so that external notifications and links use the correct host. 

app:
  base_url: "https://tasks.example.com"
Cookies and redirects
If the base URL does not match your public origin, you may see login redirects or incorrect links in emails/webhooks.

Reverse proxy notes

TimeboardApp is a standard HTTP app. A reverse proxy should:

  • Forward Host and X-Forwarded-Proto headers
  • Support WebSockets or SSE if you use browser notifications (Server-Sent Events)
  • Increase timeouts for long-lived connections if required

Example (Nginx) — illustrative only:

location / {
  proxy_pass http://127.0.0.1:8888;
  proxy_set_header Host $host;
  proxy_set_header X-Forwarded-Proto $scheme;
  proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}

Backups

SQLite backups are straightforward. Stop the container (or ensure no writes), then copy the database file from the mounted volume. For example, if you mount ./data to /data, copy ./data/timeboardapp.db.

Export/Import
The Admin UI also supports export/import workflows for migrations and recovery.

Upgrades

Recommended upgrade pattern:

  1. Backup your database and settings volume
  2. Pull the latest image (or rebuild)
  3. Start the container and review logs
  4. Validate UI + API functionality

TimeboardApp includes lightweight schema migration logic on startup.
